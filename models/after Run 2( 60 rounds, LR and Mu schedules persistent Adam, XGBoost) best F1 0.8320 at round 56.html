Root cause 1Each client sees only 3–4 of 8 classes

The partition log shows every node holds exactly 4 classes. A node that never sees dos can only learn to classify it through FedAvg aggregation, but the FedProx proximal term is actively preventing the global dos signal from fully updating that node's weights. The proximal anchor pulls local weights toward the global checkpoint — which is the correct fix for IID drift, but here it fights the signal propagation that non-dos nodes need. When mu=0.10 (run 2 phase 3), the anchor is so tight that minority-class gradients from the 4 dos nodes barely move the global model.

Root cause 236K samples over 12 nodes = ~3K each — too small for BiCNN-LSTM

The paper achieved 0.9551 training the BiCNN-LSTM on the full 36K-sample dataset centrally. In the FL setup, each client trains on ~3K samples with only 3–4 classes visible. The BiCNN-LSTM has 76K parameters — roughly 25 parameters per training sample per node. That ratio is far too high: the model is severely underfitted at the local level and FedAvg cannot compensate. The plateau at 0.83 is exactly where a 76K-parameter model saturates on 3K non-IID samples per client.

Root cause 3XGBoost baseline (0.9118) is worse than RF (0.9324)

The XGBoost replacement made the centralized baseline weaker, not stronger. RF Macro F1 = 0.9324; XGBoost Macro F1 = 0.9118. The likely cause is that XGBoost with learning_rate=0.05 and max_depth=8 overfits the majority class (benign: 60.7% of samples) when sample weights are applied uniformly, whereas RF's class_weight='balanced' handles this natively. The centralized baseline should use the best available model — RF was already that.

Root cause 4Round 2 F1 oscillation pattern is identical to run 1

The LR and mu schedules reduced oscillation amplitude slightly (±0.015 vs ±0.02) but did not break the plateau. This confirms that the problem is not optimizer dynamics — it is the data regime. Tuning optimizer hyperparameters cannot fix insufficient per-node class coverage.

Fix directionThree targeted changes that address the actual causes
FIX A — Repartition
Use 5 nodes (not 12), each with ~7.2K samples and all 8 classes. This is the NF3 minimum — sufficient for the FL argument, gives 2.4× more data per node, and guarantees every node trains on every class. Eliminates root cause 1 and 2 simultaneously.
FIX B — Restore RF baseline
Revert to RandomForest (F1=0.9324 > XGBoost 0.9118). The centralized baseline must be the strongest available model. RF was already that on this dataset.
FIX C — Reduce mu back to 0.01 fixed
With full-class 5-node partitions, client drift is no longer the dominant problem. Drop the mu schedule: keep mu=0.01 constant. The aggressive mu=0.10 in phase 3 of run 2 actively throttled minority-class gradient propagation.
